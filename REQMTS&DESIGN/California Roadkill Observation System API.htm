Account Registration

 

An account creation form is available at the following URL: http://www.wildlifecrossing.net/california/user/register. After the registration request is submitted, an email containing an initial account password is mailed to the email address specified in the registration request.

Since handling this registration process, as it stands, completely within the app is not feasible, the recommended approach is to provide a "Registration" button that launches Safari with the above URL. An explanatory note could accompany the button, instructing the use to return to the app after the registration process is complete.

Since observations submitted without an authenticated session require filling out a captcha field, the app (for now) will require account creation and authentication to operate fully.

Login

 

Once an account has been created and the password obtained, an authenticated session may be created by issuing an HTTP request with the following form

POST /california/node HTTP/1.1

Host: www.wildlifecrossing.net

Content-Type: application/x-www-form-urlencoded

Content-Length: <length>

 

name=<username>&pass=<password>&op=Log+in&form_id=user_login_block

 

For example,

POST /california/node HTTP/1.1

Host: www.wildlifecrossing.net

Content-Type: application/x-www-form-urlencoded

Content-Length: 65

 

name=seantest1&pass=mypassword&op=Log+in&form_id=user_login_block

 

If successful, an HTTP 302 response containing cookies for the authenticated session will be returned. For example,

HTTP/1.1 302 Found

Date: Fri, 24 Sep 2010 00:13:21 GMT

Server: Apache/2.2.8 (Ubuntu) PHP/5.2.4-2ubuntu5.6 with Suhosin-Patch mod_ssl/2.2.8 OpenSSL/0.9.8g

Expires: Sun, 19 Nov 1978 05:00:00 GMT

Location: http://www.wildlifecrossing.net/california/users/seantest1

The HTTP client used must support cookies, as the session cookie values must be supplied in other API requests. Note that the response may set the same cookie twice. In such a case, the later cookie value must be used.

If the Login request does not succeed, an HTTP 200 request will be returned containing an error HTML page. There is no need for the app client to parse this page. It may be assumed (for now) that an HTTP 200 response indicates that the supplied username or password is invalid.

Test results Oct 5: Supplying correct username/password credentials via NSURLConnection returns an HTTP 200 response the first time it's run, with session cookie, and a cookie expiration date 23 days in the future. Repeating the request with the same credentials returns HTTP 200 but no cookie. Removing/reinstalling the app from the device repeats first-run behavior. A first-run test with incorrect username/password pair also returns HTTP 200, with the same cookie and expiration date as if correct credentials had been provided.

Get Form Token

 

In order to submit an observation, it is necessary to obtain a "form token. A form token may be requested by issuing an HTTP request of the following form.

GET /california/node/add/roadkill HTTP/1.1

Host: www.wildlifecrossing.net

Cookie: <cookie-list>

 

 

The <cookie-list> consists of a semicolon-separated list of the cookie name-value pairs obtain from the Login response. For example,

GET /california/node/add/roadkill HTTP/1.1

Host: www.wildlifecrossing.net

Cookie: SESS013c209a2abf719c50f37613053825dd=e4562b6b2d1f9095db334948d72850e5

 

 

If successful, an HTTP 200 response containing an HTML page will be returned. The form token is present in the page as the value of the "form_token" input field of Add Observation form. This value may be extracted using the following XSL statement.

<xsl:value-of xmlns:xhtm="http://www.w3.org/1999/xhtml" select="/xhtm:html/xhtm:body//xhtm:form[@id='node-form']//xhtm:input[@name='form_token']/@value"/>

 

A complete XSL stylesheet for this operation would be as follows.

<?xml version="1.0" encoding="ISO-8859-1"?>

<xsl:stylesheet version="1.0" xmlns:xsl="http://www.w3.org/1999/XSL/Transform">

<xsl:output method="text" encoding="utf-8"/>

<xsl:template match="/">

 

<xsl:value-of xmlns:xhtm="http://www.w3.org/1999/xhtml" select="/xhtm:html/xhtm:body//xhtm:form[@id='node-form']//xhtm:input[@name='form_token']/@value"/>

 

</xsl:template>

</xsl:stylesheet>

 

If the request is unsuccessful, the form_token input field will be absent from the returned page.

 

Field Taxonomy Reference Auto-complete

 

A list of field taxonomy references may be obtained by issuing an HTTP request with the following form

GET /california/nodereference/autocomplete/field_taxon_ref/<completion string> HTTP/1.1

Host: www.wildlifecrossing.net

 

For example,

GET /california/nodereference/autocomplete/field_taxon_ref/duck HTTP/1.1

Host: www.wildlifecrossing.net

Get Change Number

 

In order to edit a previously submitted observation, it is necessary to obtain a "change number". This number appears to be used by the server to detect and prevent conflicting edits, so a new change number is required for each editing operation. A change number may be requested by issuing an HTTP request of the following form.

GET /california/node/<obs_id>/edit HTTP/1.1

Host: www.wildlifecrossing.net

Cookie: <cookie-list>

 

 

The <cookie-list> consists of a semicolon-separated list of the cookie name-value pairs obtain from the Login response. The <obs_id> is the observation ID number obtained from the trailing component of the Location header of the Add Observation response.

For example,

GET /california/node/8561/edit HTTP/1.1

Host: www.wildlifecrossing.net

Cookie: SESS013c209a2abf719c50f37613053825dd=e4562b6b2d1f9095db334948d72850e5

 

 

If successful, an HTTP 200 response containing an HTML page will be returned. The change number is present in the page as the value of the "changed" input field of Edit Observation form. This value may be extracted using the following XSL statement.

<xsl:value-of xmlns:xhtm="http://www.w3.org/1999/xhtml" select="/xhtm:html/xhtm:body//xhtm:form[@id='node-form']//xhtm:input[@name='changed']/@value"/>

 

The form token that is required for editing may also be extracted from the returned page using the following XSL statement. (The same statement as could be used to process the Get Form Token method response.)

<xsl:value-of xmlns:xhtm="http://www.w3.org/1999/xhtml" select="/xhtm:html/xhtm:body//xhtm:form[@id='node-form']//xhtm:input[@name='form_token']/@value"/>

 

If the request is unsuccessful, the form will be absent from the returned page.

Note that for most (but not all) fields, omitting the field entirely from the request will result in the field value being set to the empty string. Thus, to avoid this overwrite behavior, it is recommended that all non-image fields always be present in Update Observation requests.

The images associated with an observation may be updated using the *field_image* properties. The following image update operations are supported.

a)      Delete an image

To delete an image previously associated with an observation include the "field_image[image_number][fid]" field in the request with value 0.

b)      Add an image

To add an image, choose an image_number equal to the number of images already associated with the observation. Include "field_image[image_number][fid]" field in the request with value 0, the "files[field_image_image_number]" field containing the filename (an argument of this field) and data of the image (as the value of this field), and the "field_image[image_number][_weight]" field containing the relative importance value to be assigned to the image.

To insert a new image "between" two other images it will be necessary to adjust the importance of those two images to create a "gap" between them. Thus, the recommended approach is simply to specify a new "_weight" value for every image whenever adding a new image.

c)       Reorder images

To adjust the order of the images, for each image (except those being deleted) include the "field_image[image_number][_weight]" field containing the relative importance value to be assigned to the image.

d)      Update image descriptions

To update the description of an image, include the "field_image[image_number][description]" field containing the new descriptive text.

Note that the image_number values are always assigned starting with zero and incrementing by one according to the importance of an image, from most important to least important. If an Update Observaton request has the effect of altering the importance of images by specifying  "_weight" values for the images, the images will be renumbered as the update operation completes and will then again be numbered from 0 in order of decreasing importance.
